# Workflow Generation Behavior

This document defines how workflow files should be generated, regenerated, and preserved once workflow-file generation is implemented.

## Purpose

- make workflow-file generation behavior explicit before runtime materialization begins
- keep workflow generation aligned with the Tier D optional add-on decision
- define idempotency and overwrite expectations before any workflow files are created on disk

## Invocation Model

- workflow-file generation should be invoked through an explicit add-on or later workflow-generation path
- workflow files should not be generated by default `pa init`
- workflow files should not be silently generated by `pa init --full` under the current milestone decisions

## Optionality Rule

Workflow-file generation remains explicit add-on behavior until a later milestone deliberately changes the tier-placement decision with new evidence.

## Re-Init And Regeneration

- generated workflow files should follow the same managed-file overwrite model used elsewhere in init
- re-running generation without force should preserve existing workflow files and report skips for managed files that already exist
- regeneration with force may overwrite managed workflow files when the caller explicitly requests replacement
- regeneration should stay idempotent: running it repeatedly without content changes should not create duplicate files or alternate names

## Existing File Handling

- if a managed workflow file already exists, preserve it unless force is explicitly requested
- if an unmanaged or user-customized workflow file exists at a managed path, later implementation should surface the conflict rather than silently replacing it
- workflow generation should not create parallel variants of the same workflow file to avoid overwrite conflicts

## Unsupported Or Partial Surface Handling

- if the target workflow surface is unsupported in the current implementation, generation should stop with a clear message rather than inventing an alternate surface
- if only part of the workflow inventory is supported, generation should report that partial support explicitly
- compatibility-only workflow surfaces should remain excluded unless the caller uses a later dedicated compatibility path

## Relationship To Canonical Entry Points

- workflow generation must remain subordinate to canonical entry points and must not replace them
- regeneration must preserve that authority relationship even when workflow files are refreshed

## Reuse Contract

Later milestone 9 work should treat this document as the source of truth for invocation mode, overwrite behavior, regeneration policy, and unsupported-surface handling for generated workflow files.
